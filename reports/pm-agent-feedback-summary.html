Original feedback from beta testers on the left; team response and current status on the right.

#1 More dependencies installing the agent
Eric Wang · Suggestion
Confirmed required
Feedback

Installing the agent seems to require at least the following:

  • Pro plan for Cursor
  • GitHub access token for Cursor
  • GitHub scope added to QIMA and PM organisations
Team response

The company Cursor plan, GitHub access, and gh CLI are necessary dependencies and cannot be removed.

All other setup steps will be covered by a one-click configuration flow to reduce onboarding friction.

#2 PRD enhancement
Suki Yuan · Suggestion
Jira / Confluence limit
Feedback

The auto-generated file will be “Narrow width”; the better vision should be “Wide width”.

The requirement breakdown specified in the PRD can be refined into more concrete details by referencing the design specifications.

AI generated · Updated by me

We have attempted to change this, but it is currently blocked by Jira / Confluence API limitations.

We will continue monitoring Atlassian platform updates and adopt Wide width as soon as the API supports it.

#3 UI/UX design enhancement
Suki Yuan · Suggestion
In progress
Feedback

The design generated by PM Agent should be QIMAlabs style. There is a visible gap between AI-generated output and manually crafted design.

Team response

Among current models, only Opus 4.8 can approximate QIMAlabs style reasonably well, but limitations remain; high-fidelity frontend UI generation is costly.

We are optimising the Design Skill to reach a close approximation; manual design adjustment is still needed for each deliverable.

Validated collaboration model: PM produces prototype framework → Design refines interaction and UI details, cutting design time by roughly half.

#6 Ticket creation default setting
Genja · Suggestion
Shipped
Feedback

When creating a ticket, the agent should ask the user for preferences and a description in English by default, rather than silently writing to Jira.

Preferences to confirm:

  • Space: normally Services Platform
  • SP team: team of the PM
  • Sprint: backlog / current sprint / next sprint
  • Is BAU

Why: Each PM’s Jira context (team, sprint, space) differs. Silent defaults risk creating tickets in the wrong space or sprint, increasing PM review and correction effort.

Team response

Added to the agent fallback mechanism: when the agent cannot detect user intent or lacks required Jira context, it proactively asks in English for:

  • Space (default prompt: Services Platform)
  • SP team (PM’s team)
  • Sprint (backlog / current sprint / next sprint)
  • Is BAU
  • Preferences and description

Jira writes only proceed after the user provides or confirms these fields, avoiding silent guesses and incorrect field values.

#7 Humans and AI cannot edit the PRD simultaneously
Eric Wang · Suggestion
Under review
Feedback

When humans and AI edit at the same time, the AI overwrites the human edits.

That is, while waiting for the AI to work, humans cannot edit this PRD.

Why it matters: PMs often want to refine the PRD while the agent is still generating or updating content. Concurrent edits create a blocking workflow and risk losing manual changes.

Team response

Acknowledged. Confluence writes via the agent currently replace the full page body, so any in-flight human edits can be overwritten.

Interim guidance: Allow the agent to complete its write before opening the PRD for manual edits.

Under evaluation: Draft-first publishing, merge-safe section updates, and clearer in-agent status signalling so PMs know when it is safe to edit.
