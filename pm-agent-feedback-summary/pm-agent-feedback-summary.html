Section 02

反馈与团队回应

左侧为内测用户原始反馈，右侧为团队回应与当前状态。

#1 安装依赖过多
Eric Wang · 建议
已确认必要
用户反馈

安装 Agent 需要较多前置条件：

  • Cursor Pro 计划
  • GitHub Access Token
  • 在 QIMA 和 PM 组织中添加 GitHub scope
团队回应

公司 Cursor Plan、GitHub 权限以及 gh CLI 确实是必要依赖，无法省略。

其余配置项将由团队提供一键配置能力，降低用户上手门槛。

#2 PRD 版式为窄版
Suki Yuan · 建议
受限于 Jira
用户反馈

自动生成的 Confluence 页面默认为 Narrow width，期望改为 Wide width。

PRD 中的需求拆分可结合设计规范进一步细化。

AI 生成版 · 人工优化版

团队回应

已尝试修改，但目前受 Jira / Confluence API 限制，暂无法实现 Wide width。

将持续关注 Atlassian 平台更新，待新功能可用时第一时间适配。

#3 UI 设计风格不符 QIMAlabs
Suki Yuan · 建议
持续优化中
用户反馈

PM Agent 生成的设计应遵循 QIMAlabs 风格，当前 AI 输出与手工设计存在明显差距。

团队回应

目前市面上仅 Opus 4.8 能较好还原 QIMAlabs 风格，但仍存在局限；高质量前端 UI 生成成本较高。

团队正在优化 Design Skill，目前可做到近似还原，每次仍需设计手动调整细节。

已确立协作模式：PM 产出原型框架 → 设计优化交互与 UI 细节，设计时间节省约一半。

#4 MCP 集成：QCP 与 Microsoft 365
Eric Wang · 建议
已排期 / 转交 IT
用户反馈

需要稳定连接 QCP（目前有时报错）和 Microsoft 365。邮件沟通与会议决策是最大的上下文来源。

团队回应

Microsoft 365：已提上日程，将在后续版本支持 Cursor 直连。

QCP：连接问题将反馈给 IT 团队处理。

#5 User Story 拆分逻辑
Eric Wang · 建议 · SP-33395
工作流重构中
用户反馈
  • Epic 拆分方式与 Dev Grooming 习惯不一致
  • 错误包含 NetSuite 团队（不了解团队结构）
  • QA 工作量未合理纳入各 Ticket
  • Story Points 高估（实际 Backend 3 + Frontend 2 + QA 1）
  • 建议参考 Dev Tech Design 工作流
团队回应

整体工作流正在经历较大变革，需要与开发者协作，找到最适合 Agent 的 User Story 拆分方式。

Agent 侧改进方向：

  • 补充足够上下文，明确 NetSuite 团队工作不在内部范围
  • 让 PM 更容易审核 AI 产出的 Ticket：边界清晰、逻辑连贯、目标明确、预期落地结果具体
  • 确保 Dev Agent 能基于高质量 Ticket 直接进入开发
